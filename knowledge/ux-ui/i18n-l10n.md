# Internacionalización y localización (i18n / l10n)

> **Cuando cargar este archivo:** al preparar la UI de una app o producto para múltiples idiomas y regiones — antes de escribir el primer string, al diseñar layouts que deben aguantar traducciones más largas o de derecha-a-izquierda (RTL), al formatear fechas/números/moneda, al montar el pipeline de traducción, o al decidir a qué idiomas localizar. Es el **lado UX/producto general** (web y móvil). El **cómo técnico en un juego Unity** (Unity Localization, string tables, Smart Strings, TMP por locale, pseudo-locale, CSV) vive en [ver: pipeline/narrativa-localizacion] y no se repite aquí — este archivo da el modelo mental y las reglas que ese pipeline implementa. Priorizar idiomas por mercado: [ver: movil/mercado-movil].

Encaja entre el layout flexible de [ver: fundamentos-visuales] y [ver: mobile-ux] (reflow, expansión), los tokens de [ver: sistemas-diseno] (locale como dimensión de tema), el `lang`/reflow de [ver: accesibilidad] y el handoff de specs a dev [ver: herramientas-handoff].

**Vocabulario (abreviaturas numéricas estándar):** **i18n** = internationalization (i + 18 letras + n), **l10n** = localization (l + 10 + n), **g11n** = globalization (i18n + l10n como estrategia), **t9n** = translation, **a11y** = accessibility. Se escriben así en todo tooling y doc de la industria.

## 1. i18n ≠ l10n: dos trabajos distintos

Definiciones del **W3C** (verificadas por fetch):

| Término | Qué es (W3C) | Quién lo hace | Se hace… |
|---|---|---|---|
| **Internacionalización (i18n)** | "Diseño y desarrollo de un producto que **permite** una localización fácil para audiencias que varían en cultura, región o idioma" | Diseño + desarrollo | **Una vez**, antes (o el día 1) |
| **Localización (l10n)** | "**Adaptación** del contenido para cumplir los requisitos de idioma, cultura y otros de un mercado objetivo concreto" | Traductores + diseño | **N veces**, una por locale, y **continuo** |

- **i18n = preparar el terreno**: sacar el texto del código (strings externalizados), soportar Unicode, permitir bidi/RTL, no asumir formato de fecha/número propio, dejar el layout flexible. Es trabajo de arquitectura; se hace **antes de traducir nada**.
- **l10n = rellenar el terreno para un mercado**: traducir, formatear a las convenciones locales, adaptar imágenes/colores/ejemplos culturales.
- **La regla dura:** i18n mal hecho hace la l10n **imposible o carísima**. Traducir es fácil; re-arquitectar una app llena de strings hardcodeados, concatenaciones y anchos fijos para el inglés es un proyecto entero. **Internacionaliza desde el día 1 aunque solo lances en un idioma** — el coste de retrofitear crece con cada pantalla.
- La l10n del W3C incluye explícitamente: traducción de UI y docs; formato numérico/fecha/hora; moneda; ordenación (collation); **símbolos, iconos y colores** culturalmente apropiados; formatos de dirección y contacto; y rediseño de lógica/layout cuando las prácticas de negocio difieren.

## 2. Nunca hardcodear texto: strings externalizados + keys estables

El pecado original de la i18n es el string literal en el código o pintado en un asset. Todo texto visible sale a un **catálogo de recursos** cargado por locale.

**Catálogo por plataforma** (el formato del archivo cambia; el principio no):

| Plataforma | Formato de catálogo | Nota |
|---|---|---|
| Web / JS | JSON (i18next, FormatJS/react-intl), gettext `.po`, Fluent (`.ftl`) | `Intl` nativo para formato (§5) |
| iOS / Apple | `.xcstrings` (String Catalog, Xcode 15+), antes `.strings`/`.stringsdict` | String Catalog trae plurales y estado de traducción |
| Android | `res/values-<locale>/strings.xml`, ARB (Flutter) | Selección de recurso por `-es`, `-ar`, `-zh-rCN`… |
| Unity (juego) | String/Asset Tables + Smart Strings | Detalle: [ver: pipeline/narrativa-localizacion] |
| Interchange | **XLIFF** (estándar de intercambio con traductores) | Lo que consume un TMS/traductor |

**Keys estables — la decisión que no se deshace barato:**

- **La key NUNCA es el texto fuente.** `"Press Start"` como key: cambiar el copy rompe la key y toda traducción asociada. Usa keys **semánticas y jerárquicas**: `home.cta.play`, `settings.audio.title`, `error.network.retry`.
- La key **nace con la entrada y no se renombra** tras enviar a traducir. Renombrar = huérfanas en 20 idiomas.
- **Un concepto = una key.** No reutilices `ok` para el botón de un diálogo destructivo y el de uno informativo: el mismo inglés "OK" puede traducirse distinto según contexto y género gramatical.
- Guarda **contexto en la key o en un comentario** (quién lo dice, dónde aparece, límite de caracteres) — el traductor sin contexto adivina (§8).

**Locale identifiers (BCP 47):** `idioma-Script-REGIÓN`. `es`, `es-419` (español LatAm), `es-MX`, `pt-BR` vs `pt-PT`, `zh-Hans` (simplificado) vs `zh-Hant-TW` (tradicional Taiwán), `ar-EG`. **El idioma no basta:** `pt-BR` y `pt-PT` divergen en léxico; `zh-Hans`/`zh-Hant` son scripts distintos. Define una **cadena de fallback** (`es-MX` → `es-419` → `es` → default) para no crashear ante un locale sin traducción.

## 3. Expansión de texto: el layout tiene que estirar

El inglés es de los idiomas **más compactos**. La misma frase en otro idioma casi siempre ocupa más — y **cuanto más corto el string original, más se expande** (un botón de una palabra es el peor caso, no el mejor).

**Tabla de expansión IBM** (vía W3C, "Text size in translation" — verificada): longitud resultante como % del inglés (100% = igual tamaño):

| Caracteres en inglés | Expansión media | Implicación |
|---|---|---|
| **Hasta 10** | **200–300 %** | Un label/botón puede **2-3x** — el caso más peligroso |
| 11–20 | 180–200 % | Labels y chips |
| 21–30 | 160–180 % | Frases cortas |
| 31–50 | 140–160 % | — |
| 51–70 | 151–170 % | — |
| **Más de 70** | **~130 %** | Texto corrido: reserva **+30 %** |

- **Idiomas notoriamente largos:** alemán y finés (compuestos largos), y romances (francés, italiano, portugués, español) suelen ocupar **+30–40 %** en texto corrido; el ejemplo Flickr del W3C mostró la palabra "views" expandiéndose a 300 % en italiano, 280 % en alemán. En la dirección opuesta, CJK (chino/japonés/coreano) suele ser **más corto** pero **más alto** (glifos más densos → cuida el line-height).
- **Diseño que no rompe:**
  - **Nunca fijes el ancho/alto al texto inglés.** Contenedores que crecen (auto-layout, `min-width` no `width`, wrap permitido). En Figma: auto-layout con "hug"/"fill" (=flexbox) [ver: herramientas-handoff].
  - **Botones y labels: espacio de holgura** o truncado con tooltip; nunca cortar a media palabra sin salida.
  - **Prohibido texto en imágenes/sprites.** Texto pintado en un PNG no se traduce sin re-exportar N versiones, no escala, no lo lee un lector de pantalla y rompe el pseudo-locale. Texto = capa de texto sobre el asset, desde el catálogo. Si es inevitable (logo), trátalo como asset por locale [ver: pipeline/narrativa-localizacion §8].
  - **Prueba el peor caso ANTES de traducir** con pseudo-localización (§8): expande +40 % artificialmente y busca overflow.
- Cross-ref: la escala tipográfica, `max-width` en `ch` y el reflow a 320px de [ver: fundamentos-visuales] y [ver: mobile-ux] son la base sobre la que la expansión no debe romper; WCAG 1.4.10 Reflow / 1.4.4 Resize [ver: accesibilidad].

## 4. RTL / bidi: el layout se ESPEJA, no solo el texto

Árabe, hebreo y otros scripts se leen de derecha a izquierda. RTL **no es "alinear el texto a la derecha"**: es **espejar (mirror) toda la interfaz** — el eje de lectura completo se invierte.

**Scripts RTL (W3C):** Arabic, Hebrew, Adlam, N'Ko, Syriac, Thaana. Idiomas: árabe, hebreo, **persa/farsi, urdu**, dhivehi, yiddish. Son **bidireccionales**: dentro de texto RTL, los números y las palabras latinas (marcas, código) van LTR — el algoritmo Unicode Bidi lo resuelve, pero hay que declararlo bien.

**Qué se espeja y qué NO** (convención Material Design / HIG de bidireccionalidad):

| Se ESPEJA | NO se espeja |
|---|---|
| Layout general, columnas, navegación (drawer a la derecha) | Números (siempre LTR, "3:45") |
| Alineación de texto y bloques (a la derecha) | Reloj analógico, gráficas de tiempo con sentido físico |
| Iconos **direccionales**: flechas atrás/adelante, chevrons, "siguiente", barra de progreso, sangría de listas | Controles de **reproducción** (play sigue apuntando ⏵, es física del tape) |
| Sliders y progresos que crecen "hacia adelante" | Iconos no-direccionales (buscar, ajustes, corazón, cámara) |
| Orden de una fila de acciones / breadcrumbs | Logos, checkmarks, algunos símbolos universales |

**Cómo se hace bien (web):**
- **`dir="rtl"` en el `<html>`** (W3C: *no* usar CSS para la dirección base — es semántica del contenido, va en el markup). Eso ya realinea párrafos, tablas, inputs.
- **Propiedades lógicas de CSS, no físicas:** `margin-inline-start/end` (no `-left/-right`), `text-align: start/end`, `padding-inline`, `inset-inline`. Escrito así, el CSS **se espeja solo** al cambiar `dir`. Reescribir `left`→`right` a mano es el error clásico.
- `dir="auto"` en inputs de usuario; `dirname` para mandar la dirección al servidor.
- **Frameworks:** verifica el soporte RTL **antes de comprometer el idioma** — no todos lo dan gratis. Web moderno: sí con propiedades lógicas. iOS/Android: sí, con layouts que respetan leading/trailing (no left/right). **Unity/TMP: NO lo resuelve bien de fábrica** (shaping contextual árabe + RTL requiere plugin/trabajo extra — NO asumir soporte nativo) [ver: pipeline/narrativa-localizacion §8].
- **QA con RTL activo**, no solo leyendo: iconos al revés, texto pegado al borde equivocado, campos que empiezan a la izquierda.

## 5. Formatos locales: nunca asumas el tuyo

El formato de fecha/número/moneda/unidad **no es universal** — es una convención local, y hardcodear el propio es un bug garantizado. **No formatees a mano: usa la librería de locale** (respaldada por datos **CLDR** de Unicode).

**Lo que cambia por locale:**

| Dato | Ejemplos del mismo valor | Trampa |
|---|---|---|
| **Fecha** | `7/21/2026` (US M/D/Y) · `21/07/2026` (EU D/M/Y) · `2026-07-21` (ISO/CJK Y-M-D) | `03/04` es 3-abr o 4-mar según locale — ambiguo, evítalo |
| **Hora** | `3:45 PM` (12h US) · `15:45` (24h resto) | No asumas AM/PM |
| **Número decimal** | `1,234.56` (en) · `1.234,56` (de/es-ES) · `1 234,56` (fr) | Separador de miles y decimal **se intercambian** |
| **Moneda** | `$1,000.00` · `1.000,00 €` · `¥1,000` | Símbolo, posición, decimales y **código** (USD≠CAD≠MXN, todos "$") |
| **Primer día de semana** | Domingo (US) · Lunes (ISO/EU) · Sábado (parte de MENA) | Rompe calendarios |
| **Unidades** | km/mi, °C/°F, kg/lb, 24h/12h | Métrico vs imperial |
| **Nombres** | Given+Family (occidente) vs Family+Given (CJK/Hungría); honoríficos | **No partas "nombre/apellido" rígido**: un campo "nombre completo" es más seguro |
| **Ordenación (collation)** | ä ordena distinto en sueco vs alemán; ñ en español | No ordenes con comparación de bytes ASCII |

**El API estándar (web): `Intl`** (JS nativo, datos CLDR — verificado MDN):

| Objeto | Qué hace |
|---|---|
| `Intl.DateTimeFormat` | Fecha/hora sensible al locale |
| `Intl.NumberFormat` | Números, **moneda** (`style:'currency'`), porcentajes, unidades |
| `Intl.PluralRules` | Forma plural correcta por idioma (§6) |
| `Intl.Collator` | Ordenar/comparar strings con reglas del idioma |
| `Intl.RelativeTimeFormat` | "hace 2 días", "in 3 hours" |
| `Intl.ListFormat` | "A, B y C" con la conjunción correcta |
| `Intl.Segmenter` | Cortar en palabras/grafemas/frases (crítico en CJK/thai sin espacios) |
| `Intl.DisplayNames` / `Intl.Locale` | Nombres localizados de idioma/región; manipular el locale |

Equivalentes nativos: iOS `DateFormatter`/`NumberFormatter`/`MeasurementFormatter`; Android/Java `java.text` + ICU4J; Unity Smart Strings `{0:N}` [ver: pipeline/narrativa-localizacion §7]. **Todos leen CLDR** — la misma fuente de verdad.

## 6. Plurales y género: las reglas varían por idioma

"1 file / 2 files" (inglés = 2 formas) es la excepción simple, **no la regla**. **CLDR** define **6 categorías plurales** (verificado):

| Categoría CLDR | Para qué | Idiomas |
|---|---|---|
| **one** | singular típico (1) | casi todos los que distinguen |
| **other** | fallback obligatorio (plural general) | **todos** — el mínimo |
| **zero** | formas de cero | árabe, letón, galés |
| **two** | dual (2) | árabe, esloveno, hebreo |
| **few** | paucal (cantidades pequeñas) | ruso, polaco, checo, árabe |
| **many** | fracciones / rangos concretos | ruso, polaco, árabe |

- **Inglés: 2 formas** (`one`, `other`). **Chino/japonés/coreano: 1** (`other` — no marcan plural). **Ruso/polaco: 3–4** (`one`/`few`/`many` según terminación). **Árabe: hasta las 6.** Un `if (n === 1) singular else plural` es correcto **solo en inglés** y rompe en el resto.
- **La solución = ICU MessageFormat** (estándar de industria; Unity Smart Strings y `Intl.PluralRules` lo implementan). Mantiene el mensaje **como una sola unidad traducible con placeholders**, y cada idioma define **sus** formas:
  ```
  {count, plural, one {# archivo} other {# archivos}}
  {gender, select, female {ella} male {él} other {elle}} te espera
  ```
  El traductor de árabe añade `zero`/`two`/`few`/`many`; el de chino colapsa a `other`. Tú no tocas código.
- **La regla que salva la localización:** **jamás construyas frases concatenando fragmentos** (`"Tienes " + n + " monedas"`, `n + " item(s)"`). El orden de las palabras cambia por idioma (alemán manda el verbo al final; árabe invierte). **Una entrada con placeholders**; el traductor reordena como su gramática pida (W3C/ICU: si concatenas, el traductor no puede reordenar ni traducir los fragmentos con contexto).
- **Género:** afecta a artículos, adjetivos y verbos, no solo pronombres. `select` de ICU cubre el caso; para género del usuario, ofrece también neutro. El detalle de plural/género **dentro de diálogo de juego** (Ink/Yarn): [ver: pipeline/narrativa-localizacion §7].

## 7. Fuentes y encoding: que se vean los glifos

Traducir no sirve si la fuente no tiene el glifo: aparece **tofu** (□, el `.notdef`) o texto en blanco.

- **Encoding: UTF-8 siempre.** Unicode cubre todos los scripts; UTF-8 es el default de la web y el único encoding sano. Nada de Latin-1/legacy. Declara `<meta charset="utf-8">` y guarda los archivos en UTF-8. Encoding mal = **mojibake** (café → cafÃ©).
- **La fuente debe cubrir el script del idioma.** Una fuente "latina" no trae cirílico, griego, árabe ni CJK. Antes de prometer un idioma, **verifica que la fuente elegida tenga sus glifos**:

| Script | Coste | Nota |
|---|---|---|
| Latín ext. (á é ñ ü ç ø) + cirílico + griego | Barato | Muchas fuentes UI (Inter, Noto Sans) lo traen |
| CJK (chino/japonés/coreano) | **Caro**: miles de glifos, fuentes de **varios MB** | Subset por los caracteres usados; carga aparte; **más alto** de línea |
| Árabe / hebreo | Medio + **shaping contextual** (las letras cambian según posición) | No basta el glifo: hace falta el motor de shaping |
| Índicos (devanagari, tamil…), thai, emoji | Variable | Reordenación de vocales, clusters |

- **Fallback chain / superfamilia:** define una cadena de fuentes; si la principal no tiene el glifo, cae a la siguiente. **Noto** (Google, "No Tofu") es la familia pan-Unicode de referencia para fallback. `font-family: Inter, "Noto Sans SC", "Noto Sans Arabic", sans-serif`.
- **Peso de descarga:** una fuente CJK completa pesa MB — no la cargues en locales que no la necesitan. Carga la fuente **por locale**, no todas siempre.
- Cross-ref: en juego, atlas SDF de TMP con el character set completo del idioma o sale tofu; fuente por locale vía asset table [ver: pipeline/narrativa-localizacion §8], [ver: unity/ui-unity].

## 8. El proceso de traducción: contexto, TMS, pseudo-loc

**Qué se manda a traducir (y cómo):**

- **Nunca una lista pelada de strings.** El traductor sin contexto produce basura: "Post" ¿es sustantivo (publicación) o verbo (publicar)? "Home" ¿casa o inicio? Manda **contexto por string**: descripción, dónde aparece (screenshot ideal), **límite de caracteres**, quién habla, si es botón/título/error.
- **Formato de intercambio:** **XLIFF** (estándar) o export a **CSV/Google Sheets** para lotes pequeños (con columna de comentarios). Para escala: un **TMS** (Translation Management System — Crowdin, Lokalise, Phrase, Transifex, Weblate y similares) que gestiona keys, estados, memoria de traducción (TM), glosario y screenshots. La memoria de traducción reduce coste reusando lo ya traducido.
- **Congela los strings antes del pase final** (string freeze): cada cambio de copy **post-freeze re-abre traducción, review y QA** de esa línea en todos los idiomas.

**Pseudo-localización: probar la i18n ANTES de pagar traducción.** Genera "traducciones" falsas del idioma fuente que:
1. **Acentúan** todos los caracteres (`Ħéļļö` / `[!!! Ĥömé !!!]`) → detecta **glifos/fuentes faltantes** y qué texto NO cambia (= **string hardcodeado**, el test más barato que existe).
2. **Expanden** +30–50 % → detecta **overflow y truncado** en el peor caso sin esperar al alemán real.
3. **Encapsulan** en `[corchetes]` → detecta **strings cortados** por concatenación.

Es la verificación de i18n de mayor ROI: corre como un locale más, en cada milestone con UI nueva, **antes** de gastar en traductores. Implementación en juego: [ver: pipeline/narrativa-localizacion §9]. En web: librerías de pseudo-loc o un locale generado; en móvil, herramientas del SDK.

## 9. Culturalización: lo que significa distinto por cultura

Traducir palabras ≠ localizar significado. Colores, símbolos, gestos e imágenes **cambian de sentido** entre culturas — un acierto en un mercado puede ofender en otro.

| Elemento | Trampa | Antídoto |
|---|---|---|
| **Colores** | Blanco = luto en parte de Asia Oriental; rojo = suerte en China / peligro-prohibido en occidente; verde con connotaciones religiosas en algunos mercados | No codifiques significado crítico **solo** por color (ya es regla de a11y [ver: accesibilidad]); revisa la paleta semántica por mercado |
| **Iconos/símbolos** | Buzón de correo (US ≠ resto), papelera, ✓/✗ (el check no es universal), lechuza (sabiduría vs mal agüero), pulgar arriba (ofensivo en algunas culturas) | Iconos + texto; evita metáforas culturalmente locales |
| **Gestos / manos / dedos** | Señas de mano ofensivas según región; imágenes de manos con significado distinto | Ilustraciones neutras; probar en el mercado |
| **Números** | 4 (muerte) en CJK, 13 en occidente, 17 en Italia | No uses esos números en pisos/niveles/ratings sin pensar |
| **Banderas para elegir idioma** | ❌ **Anti-patrón:** un idioma ≠ un país. El español lo hablan ~20 países; ¿bandera de España o de México? El inglés ¿US o UK? | Lista el idioma **en su propio nombre** ("Español", "العربية", "日本語"), no con bandera |
| **Imágenes / fotos / avatares** | Vestimenta, comida, festividades, diversidad étnica, gestos religiosos | Assets neutros o localizados; evita asumir una cultura como default |
| **Formato de dirección/teléfono, sentido de lectura de flujos** | Formularios que asumen estados de US, código postal numérico, orden de campos | Campos flexibles; orden por locale |

La l10n profunda (W3C) llega a **rediseñar lógica y layout** cuando las prácticas de negocio difieren. Para un equipo pequeño: prioriza no-ofender y claridad; la adaptación cultural fina es cara y se hace solo en mercados clave.

## 10. El coste real: localizar es caro y continuo

Localizar **no es un checkbox, es un compromiso recurrente**. Cada idioma multiplica el trabajo continuo, no solo el inicial.

**De dónde sale el coste:**
- **Traducción inicial** (por palabra × idiomas) + **cada cambio de copy** re-traduce en todos los idiomas (por eso el string freeze y la memoria de traducción).
- **QA lingüístico y visual por idioma:** overflow, tofu, RTL roto, formato mal — se verifica **por locale** (la superficie de QA se multiplica).
- **VO/audio** localizado (si aplica) es el más caro; estrategia realista: VO en 1 idioma + subtítulos localizados [ver: pipeline/narrativa-localizacion §12].
- **Mantenimiento perpetuo:** cada feature nueva = strings nuevos en N idiomas para siempre.

**Qué idiomas priorizar (por mercado, no por gusto):**
- Set clásico de juegos/apps: **EFIGS** (English, French, Italian, German, Spanish) + **CJK** (Chinese Simplified, Japanese, Korean) + a menudo **Brazilian Portuguese** y **Russian**. Cubre la mayoría del gasto global con pocos idiomas.
- **Prioriza por LTV/tamaño de mercado**, no por número de hablantes: un idioma con muchos hablantes pero bajo poder adquisitivo o baja penetración de tu plataforma rinde menos que uno tier-1 [ver: movil/mercado-movil §2-3]. iOS concentra gasto en tier-1 (US, Japón, Europa occidental, Corea).
- **Palanca barata de alto retorno:** localizar **solo la ficha de tienda** (título, keywords, screenshots) a tus mercados objetivo mejora la descubribilidad orgánica sin traducir el producto entero — a veces el primer paso sensato [ver: movil/mercado-movil §4].
- **Empieza bilingüe bien hecho antes que multi-idioma mal hecho:** i18n completa + 1-2 idiomas sólidos, y añadir el tercero es coste marginal (si la i18n está bien). Lanzar 10 idiomas con la arquitectura rota es deuda que se paga en cada release.

## Reglas prácticas

1. **Internacionaliza el día 1** aunque lances en un idioma: strings externalizados, Unicode/UTF-8, layout flexible, formatos por locale. Retrofitear cuesta 10x.
2. **Cero texto hardcodeado.** Todo string visible sale a un catálogo; lo que no cambia al activar el pseudo-locale está fuera del sistema.
3. **Keys semánticas y estables** (`home.cta.play`), nunca el texto fuente como key; no renombrar tras enviar a traducir; un concepto = una key.
4. **Locale = BCP 47** (`pt-BR`≠`pt-PT`, `zh-Hans`≠`zh-Hant`) + cadena de fallback que no crashee ante locale sin traducción.
5. **Nunca fijes ancho/alto al texto inglés.** Contenedores que crecen; reserva +30 % en texto corrido y prevé 2-3x en labels cortos (tabla IBM/W3C).
6. **Prohibido texto en imágenes/sprites:** capa de texto desde el catálogo; si es inevitable, asset por locale.
7. **RTL espeja el layout, no solo el texto:** `dir="rtl"` en el markup + **propiedades lógicas** (`margin-inline`, `text-align:start/end`), no `left/right`.
8. **No todo se espeja en RTL:** flechas/chevrons/progreso sí; play, números, reloj, logos no.
9. **Nunca formatees fecha/número/moneda a mano:** usa `Intl` (web) / formatter nativo / Smart Strings — todos leen CLDR.
10. **Campo "nombre completo"**, no "nombre/apellido" rígido; no asumas orden given-family ni separador decimal.
11. **Plurales con ICU MessageFormat**, no `if (n===1)`: CLDR tiene 6 categorías (árabe hasta 6, CJK 1, ruso/polaco 3-4).
12. **Jamás concatenes frases traducidas:** una entrada con placeholders `{0}`, el traductor reordena; género con `select`.
13. **Verifica que la fuente tenga los glifos del idioma** antes de prometerlo; UTF-8 siempre; fallback a Noto; carga fuentes CJK por locale (pesan MB).
14. **Manda a traducir con contexto** (screenshot, límite de chars, rol del string), nunca una lista pelada; usa XLIFF/TMS con memoria de traducción.
15. **Pseudo-localización como gate de QA** en cada milestone con UI nueva: caza hardcode, overflow y glifos faltantes **antes** de pagar traducción.
16. **String freeze antes del QA final:** cada cambio post-freeze re-abre traducción + review en todos los idiomas.
17. **Selección de idioma por su propio nombre** ("Español", "العربية"), **nunca por bandera** (idioma ≠ país).
18. **Revisa color/símbolo/imagen por cultura** (blanco=luto, pulgar arriba, banderas); no codifiques significado crítico solo por color.
19. **Prioriza idiomas por LTV/mercado** (EFIGS+CJK+PT-BR/RU), no por número de hablantes [ver: movil/mercado-movil].
20. **QA por locale:** overflow, RTL, tofu y formato se verifican en cada idioma, no una vez en inglés.

## Errores comunes

| Pitfall | Antídoto |
|---|---|
| Strings hardcodeados "temporales" por toda la app | Catálogo desde el día 1; pseudo-locale delata lo que no cambia |
| Key = frase en inglés; cambias el copy y rompes 20 idiomas | Keys semánticas jerárquicas, inmutables tras traducir |
| Ancho fijo calibrado al inglés → el alemán se sale/trunca | Contenedores que crecen; +30 % texto corrido, 2-3x labels (tabla IBM) |
| `"Tienes " + n + " monedas"` concatenado — intraducible | Una entrada ICU con `{count, plural, …}`; el traductor reordena |
| `if (n === 1)` para plural — solo funciona en inglés | ICU MessageFormat + CLDR (árabe 6 formas, ruso 4, CJK 1) |
| Fecha `03/04` / número `1,234.56` hardcodeados | `Intl`/formatter nativo por locale (datos CLDR) |
| RTL = "alinear a la derecha" y ya | Espejar el layout: `dir="rtl"` + propiedades lógicas; QA con RTL activo |
| Reescribir `left`→`right` a mano para RTL | `margin-inline-start/end`, `text-align:start/end` — se espejan solas |
| Play/números/reloj espejados en RTL | Espejar solo lo direccional; play, dígitos y logos se quedan |
| Texto pintado en un PNG/sprite | Capa de texto desde el catálogo; asset por locale si es inevitable |
| Tofu (□) / mojibake (Ã©) al cambiar de idioma | UTF-8 + fuente con los glifos del script + fallback Noto |
| Cargar la fuente CJK (MB) en todos los locales | Fuente por locale; subset por caracteres usados |
| Mandar a traducir una hoja de strings sin contexto | Screenshot + límite de chars + rol por string; TMS con contexto |
| Cambiar una "palabrita" después del string freeze | El freeze es freeze: re-abre traducción/review/VO en todos los idiomas |
| Banderas para elegir idioma | Nombre del idioma en su propio idioma; idioma ≠ país |
| Rojo/verde o blanco sin pensar el mercado | Revisar semántica de color por cultura; nunca solo color (a11y) |
| Campo "First name / Last name" obligatorio y partido | Campo "nombre completo"; no asumas estructura de nombre occidental |
| Localizar 10 idiomas con la i18n rota | i18n completa + 1-2 idiomas sólidos primero; el 3.º es marginal |
| Elegir idiomas por nº de hablantes | Priorizar por LTV/mercado y penetración de plataforma |

## Fuentes

- **W3C Internationalization — "Localization vs. Internationalization" (w3.org/International/questions/qa-i18n)** — verificado por fetch: definiciones exactas de i18n ("diseño que **permite** localización") y l10n ("**adaptación** al mercado"); la l10n incluye formato numérico/fecha/moneda, collation, símbolos/iconos/colores culturales, formatos de dirección, y rediseño de lógica/layout. Fuente primaria del §1.
- **W3C Internationalization — "Text size in translation" (w3.org/International/articles/article-text-size)** — verificado: tabla de expansión IBM por longitud (hasta 10 chars 200–300 %; >70 chars ~130 %); ejemplo Flickr "views" (italiano 300 %, alemán 280 %). Base del §3.
- **W3C Internationalization — "Structural markup and right-to-left text in HTML" (qa-html-dir)** — verificado: scripts RTL (Arabic, Hebrew, Adlam, N'Ko, Syriac, Thaana), `dir="rtl"` en el markup (no CSS), mirroring automático de párrafos/tablas/inputs, propiedades lógicas, `dir="auto"`/`dirname`. Base del §4.
- **Unicode CLDR — "Plural Rules" (cldr.unicode.org/index/cldr-spec/plural-rules)** — verificado: las 6 categorías (zero, one, two, few, many, other); inglés 2 formas, ruso requiere 4. Base del §6. CLDR es también la fuente de datos de formato de §5.
- **MDN Web Docs — "Intl" (developer.mozilla.org)** — verificado: objetos `Intl.DateTimeFormat`, `NumberFormat` (currency), `PluralRules`, `Collator`, `RelativeTimeFormat`, `ListFormat`, `Segmenter`, `DisplayNames`, `Locale`; ejemplos `de-DE` (`26.254,39`). Base del §5.
- **ICU User Guide — "Formatting Messages / MessageFormat" (unicode-org.github.io/icu)** — verificado: MessageFormat mantiene el mensaje como unidad traducible; argumentos `plural`/`select`/`selectordinal`; por qué concatenar impide al traductor reordenar. Base del §6.
- **Material Design — Bidirectionality** — convención de qué se espeja y qué no en RTL (iconos direccionales sí; play/números/reloj no). Valor establecido de la guía; alineado con el §4 del W3C. (No re-fetch en vivo esta sesión.)
- **Base hermana `pipeline/narrativa-localizacion.md`** — el **cómo técnico en Unity** (string/asset tables, Smart Strings, pseudo-locale, TMP por locale, CSV/XLIFF, VO): este archivo da el concepto y no lo repite. Y **`movil/mercado-movil.md`** — priorización de idiomas por mercado/LTV y localización de la ficha de tienda.
